Multiple Choice Practice 1

In ROS2, what is a "node"?

🔴 Red: A physical sensor attached to the robot

🟢 Green: A single process that performs one job and communicates through topics

⚫ Black: A configuration file that launches the robot

🟠 Orange: A map cell in an occupancy grid

Answer: 🟢 Green — A single process that performs one job and communicates through topics

A node is an independent process in the ROS2 computation graph.

  • Each node does one job (e.g., lidar driver, path planner, motor controller)
  • Nodes communicate through topics (pub/sub), services, and actions
  • Nodes can be started, stopped, or replaced independently
  • A robot system is a graph of connected nodes

Command: ros2 node list shows all running nodes.

Multiple Choice Practice 2

In the ROS2 publish/subscribe pattern, which statement is TRUE?

🔴 Red: Publishers must know the identity of all subscribers

🟢 Green: Subscribers send acknowledgments back to publishers after each message

⚫ Black: Publishers and subscribers are decoupled — neither knows who the other is

🟠 Orange: Each topic can only have one publisher and one subscriber

Answer: ⚫ Black — Publishers and subscribers are decoupled

Publish/subscribe is a messaging pattern where nodes communicate through named topics without direct connections.

  • Modularity: Swap a real lidar for a simulated one — subscribers don't change
  • Scalability: Add a new node that listens to /scan without modifying existing code
  • Testing: Record and replay topic data for debugging
  • Multiple publishers and subscribers can share the same topic

Contrast with Arduino: Sensor code and motor code were tightly coupled in one file. ROS2 separates them into independent nodes connected by topics.

Multiple Choice Practice 3

In a ROS2 occupancy grid map, what does a cell value of 0 represent?

🔴 Red: An obstacle (wall)

🟢 Green: Unknown / unexplored space

⚫ Black: Free space — safe to traverse

🟠 Orange: The robot's current position

Answer: ⚫ Black — Free space — safe to traverse

An occupancy grid divides the world into cells, each storing a probability of being occupied.

  • 0 (white): Free space — the lidar confirmed nothing is there
  • 100 / 1.0 (black): Occupied — wall or obstacle detected
  • −1 or gray (?): Unknown — the robot hasn't observed this area yet

In costmaps: 0 = free, 1–252 = increasing cost, 253 = inscribed, 254 = lethal, 255 = unknown

The map is saved as a .pgm image file with a companion .yaml file specifying resolution and origin.

Multiple Choice Practice 4

What does AMCL (Adaptive Monte Carlo Localization) do?

🔴 Red: Builds a map of the environment from lidar scans

🟢 Green: Plans the shortest path from start to goal

⚫ Black: Estimates the robot's position on a known map using a particle filter

🟠 Orange: Controls the robot's wheel motors directly

Answer: ⚫ Black — Estimates the robot's position on a known map using a particle filter

AMCL answers the question: "Where am I on this map?"

  • Spreads many particles (guesses) across possible robot positions
  • Each particle is a possible (x, y, θ) pose
  • Compares each particle's expected lidar scan to the actual scan
  • Good matches survive; poor matches are removed
  • Over time, the particle cloud converges to the true pose

In RViz: Green arrows start spread out, then tighten around the real position.

Key distinction: SLAM builds the map. AMCL uses an existing map to localize.

Multiple Choice Practice 5

What is the difference between the global costmap and the local costmap in Nav2?

🔴 Red: The global costmap is for simulation; the local costmap is for real robots

🟢 Green: The global costmap covers the entire map for long-range planning; the local costmap covers a small area around the robot for real-time obstacle avoidance

⚫ Black: The global costmap uses lidar; the local costmap uses the camera

🟠 Orange: There is no difference — they are the same costmap viewed at different zoom levels

Answer: 🟢 Green — Global = full map for planning; Local = around robot for obstacle avoidance

Global Costmap:
  • Covers the entire map
  • Based on the static map you saved
  • Updated slowly (mostly static)
  • Used for long-range path planning
Local Costmap:
  • Small area around the robot
  • Updated in real-time with live lidar
  • Includes dynamic obstacles
  • Used for immediate driving decisions

Why two? Global gives the big-picture route. Local handles moment-to-moment driving and unexpected obstacles.

Short Answer Practice 1

Explain what SLAM stands for, describe the "chicken-and-egg" problem it solves, and name the key inputs and outputs of a SLAM algorithm.

First hand up gets to answer!

Sample Answer

SLAM = Simultaneous Localization And Mapping

The chicken-and-egg problem:

  • To build a map, you need to know where you are (to place each scan correctly)
  • To know where you are, you need a map (to match your sensor data against)
  • SLAM solves both problems at the same time

Inputs: Sensor data (lidar scans) + Odometry (wheel encoders)

Outputs: A map (occupancy grid) + the robot's estimated trajectory

Key techniques: Scan matching (aligning consecutive scans), loop closure (recognizing a previously visited area to correct accumulated drift).

Short Answer Practice 2

Explain how costmap inflation works and why the path planner sometimes produces a path that curves away from walls even when a straight line would geometrically work.

First hand up gets to answer!

Sample Answer

Costmap inflation expands obstacles outward by an inflation_radius.

  1. Cells on a wall → lethal (cost 254)
  2. Cells within robot_radius of wall → inscribed (cost 253)
  3. Cells within inflation_radius → decreasing cost (closer = higher)
  4. Free space beyond inflation → cost 0

Why paths curve:

  • The global planner finds the minimum-total-cost path
  • A straight line near a wall passes through high-cost (inflated) cells
  • A curved path through cost-0 cells has a lower total cost
  • The planner prefers the lower-cost route even if it's slightly longer

Effect: The robot stays safely in the middle of corridors.

Short Answer Practice 3

Compare topics and actions in ROS2. When would you use each, and what does an action provide that a topic does not?

First hand up gets to answer!

Sample Answer

Topics (Pub/Sub):
  • One-way data streams
  • Fire and forget — no response
  • Best for continuous data
  • Example: Twist to /cmd_vel
Actions (Goal/Feedback/Result):
  • For long-running tasks
  • Client sends goal; server reports progress
  • Can be cancelled mid-execution
  • Example: NavigateToPose

What actions add: (1) Feedback during execution, (2) Result on completion, (3) Goal accept/reject

Course connection: Activity 8 = topics (direct velocity). P4 Part 3 = actions (waypoint nav).

Short Answer Practice 4

Describe how to write a ROS2 Python node that publishes velocity commands to make a robot drive forward. Include the key components: node class, publisher, timer, and the message type.

First hand up gets to answer!

Sample Answer

Key components:

  1. Import rclpy, Node, and Twist (from geometry_msgs.msg)
  2. Create a class extending Node
  3. In __init__: call super().__init__('name'), create a publisher on /cmd_vel with type Twist, create a timer (e.g. 0.1s) that calls a callback
  4. In the callback: create Twist(), set msg.linear.x = 0.2, publish it
  5. In main(): rclpy.init() → create node → rclpy.spin(node) → rclpy.shutdown()

spin() is the event loop — like Arduino's loop(), it processes timer callbacks.

The Twist message has linear (x,y,z) and angular (x,y,z). For a ground robot: linear.x = forward speed, angular.z = turn rate.

Short Answer Practice 5

What is the sim-to-real gap? Give three specific examples of differences between a simulated robot and a physical TurtleBot 4.

First hand up gets to answer!

Sample Answer

The sim-to-real gap = the difference in behavior between simulation and the real world.

Three examples:

  1. Sensor noise: Sim lidar is clean. Real RPLIDAR A1 has ± a few cm noise; glass/mirrors may not reflect; dark objects may absorb the laser.
  2. Odometry drift: Sim encoders give near-perfect position. Real wheels slip on smooth floors — after 10m, position may be off 20+ cm. This is why AMCL is essential.
  3. Network latency: In Docker, instant communication. Real TurtleBot communicates over Wi-Fi with 5–50 ms delay. RViz may lag behind the real robot.

Other valid: battery drainage, auto-docking, dynamic obstacles (people), cliff sensor false positives on dark floors.

Scenario Practice 1

Your Nav2 robot keeps triggering recovery behaviors (spinning, backing up) near a narrow doorway instead of navigating through it. What could be wrong, and how would you fix it?

First hand up gets to answer!

Sample Answer

Likely problem: The inflation_radius is too large, causing inflated costmap to fill the doorway. The planner finds no path.

Diagnosis:

  • Visualize the costmap in RViz — check if the doorway is fully blocked by inflation
  • Check if robot_radius is set correctly
  • Look for the global path — if no green line appears, the doorway is blocked

Fixes:

  1. Reduce inflation_radius so the buffer fits the opening
  2. Check robot_radius — too large = inscribed zone blocks passage
  3. Reduce cost_scaling_factor for faster inflation decay

Key principle: robot_radius + inflation_radius together determine the minimum passable corridor width.

Scenario Practice 2

You need to write a ROS2 node that navigates a TurtleBot to three waypoints in sequence: (1.0, 0.0), (2.0, 1.0), and (0.0, 0.0). Describe the node structure and how you handle the transition from one waypoint to the next.

First hand up gets to answer!

Sample Answer

Structure:

  1. Create an ActionClient for NavigateToPose
  2. Store waypoints as a list, track self.current = 0
  3. Call wait_for_server() to ensure Nav2 is ready
  4. Create PoseStamped with frame_id='map' and waypoint coords
  5. Send goal with send_goal_async(goal)

Transition pattern:

  • goal_response_callback: check accepted, then get_result_async()
  • result_callback: increment index, call send_next_goal() if more remain
  • When all done, log "Route complete!"

Critical: Set frame_id='map' (not base_link). Set initial pose in RViz first. Handle rejection. Orientation uses quaternions.

Scenario Practice 3

A student's simulation map has clean, sharp walls. Their real-world TurtleBot 4 map has thick, noisy walls with gaps. What causes these differences, and how can they improve the real-world map?

First hand up gets to answer!

Sample Answer

Causes:

  1. Sensor noise: RPLIDAR A1 has ± cm variation → walls look thicker
  2. Reflective/absorbing surfaces: Glass, whiteboards, dark furniture create gaps
  3. Odometry drift: Wheel slip causes scan misalignment → blurred walls
  4. Dynamic obstacles: People walking through leave ghost traces
  5. Irregular objects: Chair legs, cables ≠ clean sim walls

Improvements:

  • Drive slowly (0.1–0.15 m/s) for better scan alignment
  • Revisit starting area for loop closure
  • Map when space is empty; keep dock away from mapping area
  • Use consistent, steady movements — avoid fast turns

Scenario Practice 4

Your robot navigates well in an open room but oscillates wildly in a long, featureless corridor. Give one localization cause and one local planner cause, and how to fix each.

First hand up gets to answer!

Sample Answer

Localization (AMCL):
  • Featureless corridor → every position looks the same
  • Particles spread out — robot is uncertain of its position
  • Fix: Increase max_particles; reduce odom_alpha if odometry is unreliable
Local planner (DWB):
  • lookahead_dist too large → overshoots corrections
  • max_angular_vel too high → aggressive turns
  • Fix: Reduce lookahead_dist and max_angular_vel; lower desired_linear_vel

Key insight: Featureless environments are hardest for both localization (no landmarks) and control (no nearby constraints). Tune for the worst case.

Scenario Practice 5

Your team has a working waypoint node in Gazebo simulation. You need to deploy it on a physical TurtleBot 4. List the key deployment steps, and identify three things that could go wrong that wouldn't happen in simulation.

First hand up gets to answer!

Sample Answer

Deployment steps:

  1. Build a real-world map (undock, SLAM, drive slowly, save map)
  2. Update waypoint coordinates for the real environment (use RViz hover)
  3. Transfer, colcon build, source workspace
  4. Set initial pose precisely in RViz
  5. Test one waypoint first before full route

Three real-world failures:

  1. Auto-docking: Robot wanders near dock → IR triggers docking → lidar stops, /cmd_vel ignored. Fix: keep dock away.
  2. Odometry drift at goal: Robot oscillates near waypoint — can't reach tight tolerance. Fix: increase tolerance to 0.3–0.5 m.
  3. Network latency: Wi-Fi delay causes timing issues with callbacks. Fix: always wait for result before sending next goal. Use ROS_DOMAIN_ID to avoid cross-talk.
